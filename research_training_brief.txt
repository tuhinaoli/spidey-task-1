INTERNAL FEASIBILITY MEMORANDUM

DATE: June 8, 2026
TO: Dr. Elena Vance, Director of ML Research Infrastructure
FROM: ML Systems Research Group
SUBJECT: Feasibility Audit and Compute Region Selection for Aegis-35B Training Run

=========================================
1. Project Scope & Objectives
=========================================
We are auditing the training parameters and compute requirements for the upcoming Aegis-35B foundation LLM. The primary goal of this audit is to identify the optimal GPU cluster setup and hosting region that minimizes training costs while staying within strict operational, financial, and environmental limits. We must choose from the available configurations in our three regional datacenters: Virginia (us-east-1), Oregon (us-west-2), and Ireland (eu-west-1).

=========================================
2. Target Model & Dataset Configuration
=========================================
- Model Parameter Count (N_params): 35,000,000,000 (35 Billion parameters)
- Dataset Size (N_tokens): 120,000,000,000 (120 Billion tokens)
- Model Layers (L): 80
- Hidden Dimension size (H): 8192
- Sequence Length (S): 4096
- Micro-batch size (B): 2

=========================================
3. Training Performance & Time Calculations
=========================================
The total training FLOPs requirement is calculated as:
    Flops_total = 6 * N_params * N_tokens

The time required to complete the training run is calculated using the Peak FP16 Tensor TFLOPS rating of the selected hardware, adjusted by the Model FLOPs Utilization (MFU) factor. MFU accounts for hardware and communication overhead during distributed training. MFU factors are architecture-dependent:
- Hopper Architecture (e.g., Nvidia H100): MFU = 42% (0.42)
- Ampere Architecture (e.g., Nvidia A100): MFU = 38% (0.38)

The training time in hours is given by:
    Time (hours) = Flops_total / (N_gpus * Peak_FP16_TFLOPS * MFU * 3600 * 10^12)

=========================================
4. Memory Footprint & Parallelism Constraints
=========================================
To train a model of this scale, we utilize Fully Sharded Data Parallelism (FSDP / ZeRO-3) to shard the model states (weights, gradients, and optimizer states) across the available GPUs. We assume AdamW optimizer is used in FP16 mixed precision. The total memory for model states is 18 bytes per parameter:
    Memory_states_per_GPU = (18 * N_params) / N_gpus

The activation memory per GPU is computed based on the micro-batch size and whether activation checkpointing is enabled. The formula for activation memory per GPU is:
    Memory_act_per_GPU = Activation_factor * L * H * B * S

Where:
- With activation checkpointing: Activation_factor = 4.5 bytes per token-layer
- Without activation checkpointing: Activation_factor = 18.0 bytes per token-layer

Critical Constraints:
- Memory Safety Buffer: The total memory footprint per GPU (Memory_states_per_GPU + Memory_act_per_GPU) must not exceed 85% of the total GPU VRAM (i.e., 68.0 GB for an 80 GB card). Exceeding this limit causes an Out-Of-Memory (OOM) training failure.
- Node Limit: Due to physical network isolation in each datacenter, the training run cannot exceed the node capacity limits specified in the datacenter rates sheet. Inter-node scaling must use standard power-of-2 GPU clusters (8, 16, 32, 64, 128, 256 GPUs).
- Cluster Size Policy: To minimize distributed training communication overhead and inter-node latency, our engineering policy requires selecting the smallest feasible cluster size (lowest number of GPUs) that satisfies the timeline constraint.

=========================================
5. Financial and Environmental Caps
=========================================
The project is governed by strict compliance bounds:
- Budget Cap: The maximum rental cost for the compute cluster is $80,000.
- Timeline Cap: Training must complete in 30 days or less.
- Carbon Offset Cap: To comply with the company's carbon-neutral pledge, total carbon emissions must not exceed 5.0 metric tons of CO2 equivalent (5,000 kg CO2e). The carbon footprint is computed using the Thermal Design Power (TDP) in Watts, the regional Power Usage Effectiveness (PUE), training time, and the local grid carbon intensity:
    Carbon_emissions (kg CO2e) = (TDP * N_gpus * PUE * Time_hours * Grid_Intensity) / 1,000,000

- Capped TDP Override: Due to a datacenter firmware update implemented on June 1, 2026, the active power draw (TDP) for all H100 SXM5 GPUs in our datacenters has been capped at 650 Watts for safety, overriding the manufacturer spec sheet value of 700 Watts. The manufacturer TDP of 700 Watts remains the spec sheet default, but for all carbon calculations in this audit, the 650 Watts capped value must be used.

=========================================
6. Audit Deliverables Required
=========================================
You are required to perform a comprehensive audit and provide two output deliverables:
- training_feasibility_sheet.csv: A CSV file containing the calculated metrics for H100 and A100 configurations in all three datacenters, specifically for standard cluster sizes (8, 16, 32, 64, 128, 256 GPUs) within their node capacity limits. The columns must include: region, gpu_model, n_gpus, time_days, total_cost, carbon_kg, memory_used_gb, is_feasible (YES/NO). Note that memory_used_gb must assume activation checkpointing is active.
- recommendation_report.docx: A formal recommendation report in Microsoft Word format. It must detail the final recommended configuration (region, GPU model, GPU count, and activation checkpointing status) and provide a detailed analysis of why all other configurations are infeasible, citing specific values, costs, memory limits, and emissions.
